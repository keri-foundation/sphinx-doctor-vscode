# Research Inbox

Use this file to capture the incoming deep-research findings before they are folded into the stable docs.

## Facts To Capture

- how Sphinx emits warnings and errors in the target workflows
- which message fields are stable enough to normalize
- what VS Code extension patterns fit the problem best
- what AI integration surface is realistic for an early slice
- what human UI pattern is most useful without overbuilding

## Decisions To Make

- how the extension should trigger or observe docs builds
- what the canonical issue record shape should be
- which UI surface ships first
- which projects or docs stacks the first milestone should support

## Risks To Watch

- output parsing that is too brittle across Sphinx versions
- UI complexity arriving before the issue model is stable
- overcoupling the AI surface to one assistant workflow
- hiding original Sphinx messages behind aggressive normalization
